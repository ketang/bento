# Version Bump

## Patch bumps are automatic

`scripts/build-plugins` computes a SHA-256 content hash of each plugin's
generated output and stores it beside the version in
`catalog/plugin-versions.json` (`{version, content_hash}`). When a plugin's
generated bytes change, its **patch** version auto-increments on the next build;
when they are unchanged, the version is left alone. You never bump patch by
hand, and there is no cosmetic-vs-behavioral judgment for patch — altered output
bumps, unaltered output does not.

The only obligation: after any change under `catalog/`, run
`scripts/build-plugins` and commit its regenerated outputs (`plugins/`,
`.claude-plugin/marketplace.json`, `catalog/plugin-versions.json`) together. CI's
`scripts/build-plugins --check` fails otherwise.

## When to use this skill

Only to decide whether a change warrants a **minor** or **major** bump instead
of the default automatic patch. Content hashing cannot tell a backward-compatible
addition from a breaking change; that intent call is yours.

## Choose the semver part

**Minor** — backward-compatible additions of user-visible capability:
- Adding a new skill to an existing plugin (absent a compatibility break)
- New optional workflow paths, commands, hook points, customization files, or
  generated output fields
- Expanded support for another runtime, tracker, framework, or repo convention

**Major** — intentional compatibility breaks or significant contract changes:
- Removing or renaming skills, hooks, commands, customization paths, manifest
  fields, generated artifacts, required inputs, expected outputs, or lifecycle
  order
- Changing lookup precedence or canonical source locations users must adapt to
- Removing support for a runtime or previously supported convention

If neither applies, do nothing — the automatic patch bump already covers it.
When a change matches both, use the higher part.

## Apply a manual minor/major bump

1. Edit the `version` field for the affected plugin(s) in
   `catalog/plugin-versions.json` to the intended minor- or major-bumped value.
   Leave `content_hash` as-is.
2. Run `scripts/build-plugins`.
   - If the content is otherwise unchanged, the build keeps your version
     verbatim and just regenerates the manifests.
   - If content also changed, the automatic patch increment rides on top of your
     manual major/minor (e.g. a manual `2.0.0` with a content change becomes
     `2.0.1`); the major/minor you set is never lowered.
3. Commit the version file with the regenerated artifacts together:
   ```
   git add catalog/plugin-versions.json plugins/ .claude-plugin/marketplace.json
   git commit -m "chore: bump <plugin> to <version>"
   ```

## Constraints

- Do not hand-edit generated manifests under `plugins/` or
  `.claude-plugin/marketplace.json`; regenerate via `scripts/build-plugins`.
- Pushes are independent of version bumps — push eagerly and frequently.
